## 1. 产品概述

### 1.1 一句话定位

一个**本地优先、双端可用、AI 辅助理解**的开源电子书阅读器：用户把书放进本地书架，在阅读中遇到看不懂的概念时，AI 用通俗语言和贴合用户自身生活/使用习惯的例子把它讲清楚。

### 1.2 产品愿景

大多数阅读器解决的是「把字显示出来」，而真正的阅读障碍常常发生在**理解层**——一个陌生的术语、一段绕的论证、一个脱离读者经验的抽象概念，会让人卡住甚至弃读。本产品的核心主张是：**阅读器不该只翻页，还该在你卡住的那一秒帮你跨过去**，并且这个「帮」是个性化的——同一个概念，对程序员和对厨师，应该给不同的类比。

### 1.3 与现有产品的差异

| 维度 | 传统阅读器（如 Calibre / 各家 App） | 本产品 |
|---|---|---|
| 书籍来源 | 用户自行导入 / 平台商店 | 用户导入 + 合法公版源聚合搜索 |
| 核心体验 | 排版、翻页、标注 | 排版翻页标注 **+ AI 概念解释** |
| 解释方式 | 无 / 词典式释义 | 通俗化 + **个性化类比**（结合用户画像） |
| 数据归属 | 云端账户 | **本地优先**，书架存本地 |
| 商业属性 | 商业闭源为主 | 开源、非商业 |

**护城河判断**：书籍获取和阅读器本身是红海，任何团队都能做；真正难以复制、也最有价值的是 **「上下文感知 + 个性化类比」的 AI 解释体验**。资源应向这里倾斜。

### 1.4 目标用户

- **第一用户：作者本人**。设备为 macOS（Apple Silicon）+ Android 手机；中英文书都读，英文书需要中译辅助才能读懂。产品所有决策以「先把作者自己的阅读闭环跑通」为最高优先级。
- **延伸用户**：有一定阅读量、会主动啃硬书（社科、科普、专业、经典）但常被概念卡住的人群——学生、自学者、跨领域读者、非母语读者。
- **社区用户**：注重数据隐私、偏好本地存储和开源工具的技术人群（也是开源社区早期贡献者的重要来源）。

### 1.5 核心价值主张

1. **看得懂**：卡在概念上时即时获得贴合自己经验的解释，降低弃读率。
2. **随处看**：电脑起、手机续，进度和标注在双端一致。
3. **自己的书、自己的数据**：书架本地存储，阅读数据不被平台占有。
4. **开放**：开源、可自建、数据源可插拔。

## 2. 范围与非目标

### 2.1 本期范围（In Scope）

- 双端（桌面 + 移动）电子书阅读器，支持主流格式。
- 本地书架管理。
- 合法公版书搜索与导入。
- 中立的书籍聚合搜索入口（可插拔数据源）。
- AI 概念解释（选段触发、通俗化、个性化类比）。
- 双端进度/标注同步。

### 2.2 非目标（Out of Scope，本期不做）

- 不做在线书城 / 交易 / 会员付费体系。
- 不做社交、书评社区、UGC 广场。
- 平台**不内置针对特定盗版站点的定向抓取或下载代理**（见 §8 合规）。
- 不做有声书 TTS 合成（可作为远期）。
- 不做 OCR 扫描版精排（可作为远期）。

## 3. 使用场景（User Stories）

1. **啃硬书的学生**：读《资本论》读到「使用价值 / 交换价值」绕不清，选中段落 → AI 结合她网购比价的日常，用「同一件羽绒服，对你保暖是使用价值，能卖 300 块是交换价值」讲清楚。
2. **跨行业自学者**：产品经理读一本讲统计的书，遇到「置信区间」，AI 知道他是做产品的，用「A/B 测试结果的波动范围」来类比，而不是用抛硬币。
3. **通勤续读者**：早上在家用电脑读了两章并划了线，地铁上掏出手机，进度和划线都在，接着读。
4. **隐私敏感用户**：把自己合法持有的 epub 拖进书架，全程本地存储，只有触发 AI 解释时才有一次明确的、可关闭的网络请求。
5. **找公版经典的人**：想读《国富论》，在应用内搜索 → 命中 Standard Ebooks / 维基文库的合法版本 → 一键导入书架。

### 4.4 AI 概念解释（护城河模块）

| 编号 | 功能 | 说明 | 优先级 |
|---|---|---|---|
| D1 | 选段触发解释 | 用户划选词/句/段 → 弹出「解释」动作 | P0 |
| D2 | 通俗化解释 | AI 用尽量简单的语言解释所选概念，避免以术语解释术语 | P0 |
| D3 | 上下文感知 | 解释时携带当前书名、章节、所选段落的前后文，保证解释贴合本书语境而非泛泛而谈 | P0 |
| D4 | 个性化类比 | 依据用户画像（职业、兴趣、日常场景）生成贴合其经验的例子；同一概念对不同用户给不同类比 | P0（差异化核心） |
| D5 | 追问/对话 | 对解释不满意可继续追问，形成局部对话 | P1 |
| D6 | 解释深度切换 | 「一句话 / 展开 / 举例 / 更深」多档深度 | P1 |
| D7 | 用户画像管理 | 用户可设置并随时编辑职业、兴趣、偏好的类比风格；也可选择「不个性化」 | P0（与 D4 配套） |

| D9 | 术语一致性 | 同一本书内对同一术语的解释保持一致（缓存/记忆） | P2 |
| D10 | 生词/概念本 | 自动沉淀查过的概念，形成个人概念本 | P2 |

### 4.4b AI 翻译辅助（英文书中译）

| 编号 | 功能 | 说明 | 优先级 |
|---|---|---|---|
| G1 | 选段即时翻译 | 划选英文段落 → 即时中译，与「解释」并列出现在浮起工具条中 | P1 |
| G2 | 全书批量翻译 | 后台任务将整本英文书翻译为中文对照版；**默认走 Mac 本地 Ollama，零 API 成本**，可挂机（如夜间）执行；进度可暂停恢复 | P1 |
| G3 | 双语对照阅读 | 原文/译文段落级对照或切换显示 | P2 |
| G4 | 翻译质量预期管理 | UI 明示本地模型译文为「辅助理解级」，非出版级；可选用 API 模型重译关键章节 | P1（随 G2）|

> 定位说明：翻译服务于「看懂」而非「出版」。本地 7B–14B 模型的译文质量足以辅助理解，成本为零；对质量敏感的段落，用户可临时切换 API 模型重译。

## 5. 交互流程（关键路径）

### 5.1 首次使用

启动 → 选择界面语言 → （可选）填写用户画像（职业/兴趣/类比偏好，可跳过）→ （可选）配置 AI 服务 → 进入空书架，引导「导入本地书」或「搜索公版书」。

### 5.2 核心：AI 解释一次交互

阅读中划选文本 → 浮起工具条（高亮 / 笔记 / **解释**）→ 点「解释」→ 系统组装上下文（书名 + 章节 + 选段前后文 + 用户画像）→ 调用 AI → 底部抽屉展示「通俗解释 + 贴合你的例子」→ 用户可【换个例子】【更深入】【追问】【存为笔记】。

### 5.3 公版书获取

书架空态或搜索页输入书名 → 命中合法源结果卡片（来源、语言、格式、许可标注）→ 一键导入 → 入本地书架。

## 6. 技术架构建议

> 以下为建议方案，非强制。核心诉求：**一套代码尽量覆盖双端、本地优先、AI 层可插拔**。

### 6.1 整体分层

```
┌─────────────────────────────────────────────┐
│  UI 层（双端共享的阅读器 / 书架 / 解释抽屉）      │
├─────────────────────────────────────────────┤
│  应用逻辑层                                     │
│   · 书架管理   · 阅读器引擎   · 标注/进度         │
│   · 解释编排器(Explain Orchestrator)           │
│   · 数据源适配器(Source Adapters)              │
├─────────────────────────────────────────────┤
│  本地数据层（SQLite + 文件存储）                 │
├─────────────────────────────────────────────┤
│  外部依赖（按需、可关闭）                         │
│   · AI 模型 API（用户配置）                      │
│   · 合法公版源 / 元数据 API                      │
└─────────────────────────────────────────────┘
```

### 6.2 技术选型候选

| 关切点 | 方案（已定/建议） | 说明 |
|---|---|---|
| 跨端 UI | **Flutter（已定）**，目标平台 macOS + Android | 一套代码双端；AI 辅助开发生态成熟，示例代码丰富，适合「AI 帮写」的开发方式 |
| EPUB 渲染 | Flutter 侧 epub 渲染库（如 flutter_epub_viewer / 基于 WebView 的 epub.js 封装） | Phase 0 技术验证的重点项 |
| 本地存储 | SQLite（元数据、标注、进度）+ 应用目录文件（书籍原文件） | 本地优先的基石；库表设计需 Syncthing 友好（见 E4） |
| 全文检索 | SQLite FTS5 | 书内/书架检索 |
| AI 接入 | 抽象 `LLMProvider` 接口。**默认：Ollama**（Mac 本地 / Android 走局域网连 Mac）；适配 OpenAI 兼容 API / Anthropic；端上小模型为 P2 | Apple Silicon 跑 7B–14B 模型（如 Qwen2.5 级别）满足解释与翻译的「看懂级」质量，零 API 成本 |
| 同步 | **Syncthing（已定）**：同步书架目录 + 元数据导出文件；App 内不实现同步协议，只做「Syncthing 友好」的数据布局与冲突合并 | 零服务器、点对点、开源；WebDAV 适配器为 P2 |

### 6.3 「解释编排器」设计（核心组件）

这是护城河功能的技术承载，职责是把一次「解释」请求组装成高质量的模型调用：

1. **上下文采集**：所选文本 + 前后 N 段落 + 书名/作者/章节标题。
2. **用户画像注入**：职业、兴趣、偏好类比领域、期望深度（来自 D7 设置）。
3. **提示词模板**：系统提示约束「用通俗语言、避免以术语解释术语、给一个贴合用户画像的具体例子、必要时指出该概念在本书语境下的特定含义」。
4. **深度/换例控制**：不同档位与「换个例子」映射为不同的追加指令，复用同一上下文。
5. **一致性缓存**（P2）：对同书同术语的解释做本地缓存/记忆，保证 D9。
6. **隐私边界**：仅在用户显式触发时外发；外发内容对用户可见、可关闭（走本地 Ollama 时数据不出设备/局域网）。

### 6.3b 翻译管线（G1/G2 的技术承载）

- **选段翻译（G1）**：与解释共用 `LLMProvider`，轻量提示词，即时返回。
- **批量翻译（G2）**：桌面端后台任务队列。将 EPUB 按章节→段落切分，逐段送本地 Ollama 翻译，写入平行的译文库（保留原文段落 ID 映射，支撑 G3 对照）；任务可暂停/恢复/断点续跑；完成后生成中文版或双语版 EPUB 存入书架。
- **成本模型**：本地模型批量翻译零 API 成本，代价是时间（一本 300 页英文书在 M 系列 Mac 上约数小时量级，可夜间挂机）；关键章节可选 API 模型重译（G4）。

### 6.4 数据源适配器设计（合规关键）

- 定义统一接口 `BookSource { search(query), fetch(id) }`。
- **随包默认只注册合法源**（Gutenberg、Standard Ebooks、维基文库等）。
- 适配器可由用户在设置中增删；平台代码库不包含任何盗版站定向适配器。
- 该设计使「工具中立性」在架构层面可验证，也便于社区在合法边界内扩展。

### 6.5 本地数据模型（简化）


## 7. 非功能需求

- **性能**：书架 500 本内秒开；翻页无感延迟；AI 解释首字返回目标 < 3s（本地 Ollama 于 Apple Silicon 上可达；局域网/API 视网络）。
- **离线**：除 AI 解释、公版源搜索外，全部功能离线可用（E5）。
- **隐私**：默认本地存储；任何外发（AI、搜索、元数据）均可见、可关闭；不采集用户阅读行为上报（F2/F3）。
- **安全**：API Key 存于系统 Keychain/Keystore；不随书架备份导出。
- **可移植/开源友好**：清晰的模块边界、文档化的适配器接口，降低社区贡献门槛。
- **可访问性**：字号无极限缩放、高对比主题、屏幕阅读器基本支持。

## 8. 合规与风险（务必立项前评估）

> 免责声明：以下为产品设计层面的风险梳理，**不构成法律意见**。正式立项/发布前建议咨询专业律师。

### 8.1 设计层面已采取的风险控制

- **书架纯本地存储**：平台不托管、不缓存用户导入的任何书籍文件，规避「平台存储侵权内容」的直接责任。
- **不内置盗版定向抓取**：随包数据源仅合法公版源；聚合搜索为中立入口，下载动作在用户侧、外部完成。
- **数据源可插拔且默认合法**：用户自行添加的源属其个人配置行为，责任边界清晰。
- **合规标注贯穿功能清单**：每个获取类功能标注 🟢/🟡/⚪，责任归属显性化。

### 8.2 仍然存在、需你知悉的风险点

- **「设链/索引」的帮助侵权风险**：即便不经手文件，若某功能被认定为「明知指向侵权内容仍提供便利」，在多数司法辖区仍可能构成帮助/诱导侵权。中立性设计能降低但**不能完全消除**该风险，尤其当默认或推荐的源指向侵权内容时。→ 控制手段：默认源保持合法、不对盗版源做推荐/置顶/预置。
- **开源托管平台下架风险**：GitHub 等平台对 DMCA/侵权投诉响应积极，项目知名度上升后被投诉下架的概率增加。→ 控制手段：README/功能设计中明确合法用途定位，避免以「找盗版书」作为卖点宣传。
- **元数据/封面版权**：抓取封面、简介需使用允许再分发的来源（如 Open Library 开放数据）。
- **AI 外发内容**：向第三方模型发送书籍片段，需在隐私说明中告知用户，避免误发敏感/受限内容。

### 8.3 建议的立项前动作

1. 就「聚合搜索/可插拔源」的具体形态咨询律师，确认默认源与 UI 呈现方式的边界。
2. 明确项目对外定位为「合法阅读 + AI 理解辅助工具」，宣传口径不指向盗版获取。
3. 准备 DMCA/侵权响应流程（开源项目也应有）。

## 9. 开发路线图

原则一：**先把护城河跑通，再补齐周边**——工程资源集中到 AI 解释/翻译体验，不在红海功能上过度投入。
原则二：**适配「AI 辅助开发」的节奏**——每个阶段切成小而完整的可验证增量（一次让 AI 实现一小块、当场跑通验收），避免一次性生成大工程导致失控；里程碑验收标准全部写成「作者可亲手操作验证」的行为描述。

### Phase 0 — 技术验证（Spike，1–2 周）

目标：验证最不确定、风险最高的三件事。

- Flutter 跑通「macOS + Android」双端最小骨架（一个能翻页显示 EPUB 的空壳）。
- 打通端到端「选段 → 组装上下文 → 调本机 Ollama → 展示解释」链路（哪怕 UI 很粗）。
- 验证 Android 通过局域网调用 Mac 上 Ollama 的连通与延迟。
- 产出：三条链路的可行性结论；EPUB 渲染库选型定稿。

### Phase 1 — MVP（macOS 单端可用闭环，约 4–6 周）

目标：作者能真正用它读一本书并被 AI 帮到。**只做 P0，只做 macOS 端**。

- 本地导入 epub/txt（A1）+ 本地书架（B1/B2）。
- EPUB 阅读器：渲染、排版、主题、进度、划线（C1–C5、C9）。
- AI 解释核心：选段触发、通俗化、上下文感知、个性化类比、画像设置（D1–D4、D7）。
- AI 配置（默认检测本机 Ollama）与隐私开关（F1/F1b/F2/F3）。
- 里程碑验收：**用自己的画像在 Mac 上读一本硬书，AI 给出的例子确实贴合我的经验，全程零 API 费用**。

### Phase 2 — V1（Android 端 + 同步 + 翻译，约 6–8 周）

- Android 端补齐（E2，移动交互模式：长按划选、底部抽屉、沉浸翻页）。
- Syncthing 友好的数据布局 + 双端进度/标注同步（E3/E4/E5）。
- Android 连接局域网 Ollama / API Key 双通道（F1b）。
- **翻译辅助上线**：选段即时翻译（G1）、全书批量翻译（G2）、质量预期管理（G4）。
- 公版书搜索与一键导入（A2/A3/A6，中文侧维基文库等、英文侧 Gutenberg/Standard Ebooks）。
- AI 解释增强：追问、深度切换、存为笔记（D5/D6/D8）。
- 里程碑验收：**早上 Mac 上读的进度和划线，出门后手机上都在；一本英文书挂机一晚变成能看懂的中文版**。

### Phase 3 — V2（体验完善 + 生态，持续迭代）

- 笔记书签、书内检索、PDF 阅读（C6–C8）；书架分组标签、排序检索（B3/B4）。
- 双语对照阅读（G3）。
- 中立聚合搜索入口 + 可插拔数据源（A4/A5）。
- WebDAV 等其他同步适配器（E4 扩展）；术语一致性、概念本（D9/D10）。
- 多语言 UI（F4）、书架导入导出（B5）。
- 社区方向：Windows/Linux/iOS 端、数据源适配器、AI Provider、主题的插件化。

### 优先级速览

| Phase | 关键交付 | 对应功能 |
|---|---|---|

## 10. 决策记录与剩余开放问题

### 已定决策（2026-07-18 对齐）

| 事项 | 决策 | 依据 |
|---|---|---|
| 跨端框架 | Flutter | 一套码覆盖 macOS+Android；适合 AI 辅助开发 |
| 目标平台 | macOS（Apple Silicon）+ Android | 作者自有设备；其余平台交社区 |
| 首发端 | macOS 先行，Android 于 Phase 2 | 桌面端开发调试效率高，本地 Ollama 就在 Mac 上 |
| AI 模型策略 | 默认本地 Ollama（Mac 本地/手机走局域网），API Key 可选替换 | 零成本起步；Apple Silicon 性能足够 |
| 双端同步 | Syncthing | 无服务器、零成本、开源 |
| 部署形态 | 无自建服务器，纯本地 + GitHub 开源分发 | 作者无服务器 |
| 翻译需求 | 新增 G 组功能（选段翻译/批量翻译/对照） | 作者读英文书需中译 |

### 剩余开放问题

1. **个性化画像粒度**：用户画像做到多细？（仅职业/兴趣，还是允许自由文本描述生活场景？建议 MVP 用「职业 + 兴趣标签 + 一段自由描述」三件套。）
2. **批量翻译的默认模型**：Ollama 下选哪个模型做翻译默认（候选：Qwen2.5 系列对中文最友好），Phase 0 实测后定。
3. **开源许可证**：GPL 系（衍生必须开源）还是 MIT/Apache（宽松）？影响社区形态，发布前定即可。
